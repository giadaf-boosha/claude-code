# 15b — Planning strategy

> 📍 [README](../README.md) → [Concetti foundation](../README.md#concetti-foundation) → **15b Planning strategy**
> 📘 Concettuale · 🟡 Intermediate

> **Tesi del capitolo**: il pilastro **Planning** del framework IMPACT non e' "preparare un piano dettagliato sempre". E' decidere **quando pianificare e quanto**, in funzione della complessita' e del costo di errore. Claude Code offre 4 livelli di planning in spettro continuo: zero plan → plan mode → `/ultraplan` → `/batch`. Sapere quando salire/scendere lo spettro vale piu' di sapere come usare ognuno.

---

## 15b.1 Perche' "Planning" e' un componente IMPACT a se'

L'agent loop ReAct (vedi [14b](./14b-agent-loop-react.md)) interleava Reason + Act + Observe. Ma su task complesso, **fare Reason solo turn-by-turn non basta**: l'agent puo' partire male e correggere a costo alto.

**Planning esplicito = Reason concentrato all'inizio**, prima del primo Act. Il piano e' un asset intermedio (markdown, browser review) che:
1. Rende esplicito l'approccio
2. Permette correzione a costo zero (vs fix di codice gia' scritto)
3. Diventa Memory persistente (plans directory)
4. Distribuisce il "thinking" su un modello reasoning-heavy (Opus 4.7) lasciando l'execution a un modello veloce (Sonnet 4.6)

| Livello | Comando | Durata plan | Modello consigliato | When |
|---|---|---|---|---|
| 0 — Zero plan | (nessuno) | 0 min | Sonnet 4.6 | Bug fix, edit isolato, prototyping |
| 1 — Plan mode locale | `Shift+Tab` o `/plan [task]` | 1-5 min | Opus per Reason | Cross-file ma 1 feature, durata 30min-2h |
| 2 — Ultraplan cloud | `/ultraplan <prompt>` | 10-30 min | Opus 4.7 xhigh | Refactor cross-modulo, design feature complessa, durata 1-3 giorni |
| 3 — Batch | `/batch <instruction>` | Plan + interview | Subagent worktree array | Migrazione 100+ unit, code-mod massivo |

## 15b.3 Quando salire/scendere lo spettro

### Salire (piu' planning)
- Cross-file coupling alto (1 cambio rompe N file)
- Costo errore alto (production, db migration)
- Tempo richiesto > 2h
- Approval umana richiesta (PR pubblica, design)
- Necessita' di review prima di Act (compliance)
- Modello necessita ragionamento esteso (algoritmo nuovo)

### Scendere (meno planning)
- Task ripetitivo gia' fatto 100 volte
- Bug evidente (linter rosso, errore puntuale)
- Prototipo throw-away
- Tu hai gia' il piano in testa, l'agent solo esegue
- Time-pressure massimo, costo errore basso (test env)

## 15b.4 Pattern operativi

### Pattern A — "Plan once, execute many"
Per task da spalmare su giorni:
1. `/ultraplan` cloud → piano markdown approvato
2. `claude --resume <session-id>` riprende quando torni
3. `/plans/<auto-generated-name>.md` resta come asset versionato

> Plan = Memory di lunga durata. Vedi [06b — Memory architecture § Layer 5](./06b-memory-architecture.md).

### Pattern B — "Plan locally, execute remotely"
Pattern team Anthropic interno:
1. CLI per pensare il problema, scrivere prompt + constraint
2. `/ultraplan` shippa in cloud (plan in browser, review iterativa)
3. "Approve and start coding" → cloud execute, PR aperto
4. "Approve and teleport back to terminal" → local execute

> Vedi [@noahzweben](https://x.com/noahzweben/status/2037219115002405076) — "Cloud Auto-Fix in cloud".

### Pattern C — "Opus per plan, Sonnet per execution"
Tip Cat Wu (ago 2025): un setting ha ancora questa logica.
1. `/model` Opus per plan mode
2. Sonnet 4.6 per esecuzione
3. Token saving + best-of-both

> [@_catwu](https://x.com/_catwu/status/1955694117264261609)

### Pattern D — "Plan then split"
Per refactor ampi:
1. Ultraplan produce piano cross-modulo
2. Plan diviso in N batch indipendenti
3. `/batch <instruction>` distribuisce a worktree paralleli
4. PR multipla, review distribuita
5. Merge sequenziale

> Vedi [@bcherny](https://x.com/bcherny/status/2038454355469484142) — `/batch` fan-out.

### Pattern E — "Pre-mortem planning"
Prima di Act, l'agent deve ragionare:
- Cosa puo' andare storto?
- Che rollback ho?
- Che test ho per sapere se ha funzionato?

Boris tip 13: "Give Claude a way to verify its work" → 2-3x quality.
Vedi [@bcherny](https://x.com/bcherny/status/2007179861115511237).

## 15b.5 Anti-pattern di planning

| Anti-pattern | Conseguenza | Fix |
|---|---|---|
| Plan mode su bug a 2 righe | Friction inutile | Skip plan, edit diretto |
| Skip plan su refactor cross-file | Rifare 3x | Plan mode obbligatorio sopra soglia |
| Ultraplan cloud per task da 30 min | Overkill, costo extra usage | Plan mode locale basta |
| Plan dettagliato ma poi execute "a feeling" | Plan irrealizzato, ricominci | Implement here / new session disciplinato |
| Mai ripristinare plan via `--resume` | Perde piani approvati | `claude --resume <name>` |
| `/batch` su < 5 file | Overhead worktree spawning | Edit normale |

## 15b.6 Costo plan vs costo fix

Modello mentale (semplificato):

```
Costo totale = Costo planning + Costo execution + Costo fix errori

dove:
  Costo planning ~ tempo Opus * effort
  Costo execution ~ tempo Sonnet * iterazioni
  Costo fix errori ~ (probabilita' errore * costo blast radius) - costo evitabile con plan

Soglia di convenienza plan:
  pianifica se Costo planning < (probabilita' errore * costo fix)
```

In pratica: piu' alto e' il blast radius (production, multi-team), piu' conviene pianificare.

## 15b.7 Approval loop e revisioni

Plan mode in Claude Code finisce con un dialog:
1. **Yes, implement here** (esce plan mode, esegue)
2. **No, refine with Ultraplan** (escala a cloud)
3. **No, give feedback** (continua iterazione)

Ultraplan in browser ha:
- Inline comment su passaggi specifici
- Emoji reaction
- Outline sidebar
- "Refine" iterativo
- "Approve and start coding" / "Approve and teleport back" / Cancel

Questo e' **planning con review umana** integrata. Vedi [15 — Ultraplan & Ultrareview](./15-ultraplan-ultrareview.md).
